# Model Card

Veridian ships two supervised classifiers. Both are evaluated on a held-out test
set against a majority-class baseline, and the numbers below are reported as
measured — no inflation. Metrics are regenerated by `python -m models.train` and
written to `reports/` (`metrics_delay.json`, `metrics_low_review.json`,
`model_report.md`, confusion matrices, and SHAP plots).

## Models at a glance

| | `delay` | `low_review` |
|---|---|---|
| Task | Predict an order is delivered later than the customer's estimate | Predict a customer leaves a 1–2★ review |
| Label | `is_late` (delivered after estimate) | `low_review` (review score ≤ 2) |
| Framing | Order-time (pre-outcome) | Post-delivery |
| Modeled population | Delivered orders | Delivered orders with a review |
| Test base rate | 8.1% positive | 12.8% positive |
| Algorithm | XGBoost in a scikit-learn pipeline, isotonic-calibrated | Same |

## Data

- **Source:** Olist Brazilian E-Commerce dataset (~100k orders, 2016–2018). The
  full entity model and column definitions are in
  [`data_dictionary.md`](data_dictionary.md).
- **Modeling frame:** the `order_features` warehouse table joined to the delivery
  outcome columns of `orders_order_level`. Both models train on **delivered
  orders only** (96,478 orders), since delivery and review outcomes are only
  defined for delivered orders.
- **Split:** 64% train / 16% validation / 20% test, stratified on the label.
  - `delay`: 61,740 train · 15,436 validation · 19,294 test.
  - `low_review`: 61,332 train · 15,333 validation · 19,167 test (orders with a review).

## Features

Feature sets are defined once in `models/features.py` and shared by training, the
saved metadata, and the API request schemas so they cannot drift.

**Order-time features (both models):**

- *Numeric:* `estimated_delivery_days`, `approval_delay_hours`, `n_items`,
  `n_sellers`, `n_products`, `total_price`, `total_freight`, `freight_ratio`,
  `avg_product_weight_g`, `max_product_weight_g`, `avg_product_volume_cm3`,
  `max_installments`, `n_payments`, `customer_seller_distance_km`,
  `purchase_month`, `purchase_dow`, `purchase_hour`, `purchase_is_weekend`,
  `cross_state_shipment`.
- *Categorical:* `main_category`, `primary_payment_type`, `customer_state`,
  `main_seller_state`.

**Additional post-delivery features (`low_review` only):**
`actual_delivery_days`, `delivery_vs_estimate_days`, `is_late_int`.

## Methodology

- **Preprocessing.** A `ColumnTransformer` applies median imputation to numeric
  features and most-frequent imputation plus one-hot encoding (with
  `handle_unknown="ignore"` and a minimum category frequency) to categoricals.
- **Estimator.** `XGBClassifier` (300 trees, max depth 5, learning rate 0.08,
  subsample/colsample 0.9, L2 regularization), with `scale_pos_weight` set from
  the training-set class ratio to handle imbalance.
- **Threshold tuning.** The operating threshold is chosen to maximize F1 on the
  **validation** split only. The test set is never used for threshold selection,
  so reported test metrics are unbiased.
- **Calibration.** The serving model wraps the pipeline in
  `CalibratedClassifierCV` with isotonic regression (3-fold), so the API returns
  usable probabilities rather than raw scores. Calibration roughly halves the
  Brier score (see below).
- **Baseline.** Every model is compared against a majority-class predictor, which
  scores ROC-AUC 0.5 and 0 recall on the positive class.

### Leakage control

The leakage boundary is enforced through the per-model feature lists, not
convention:

- The **`delay`** model may use only features known at or shortly after purchase.
  It never sees the delivery outcome it predicts.
- The **`low_review`** model is framed as *post-delivery*: the customer's review
  is created after the order arrives, so delivery-outcome features
  (`actual_delivery_days`, `delivery_vs_estimate_days`, `is_late_int`) are
  legitimate inputs, not leakage.

## Held-out metrics

Measured on the 20% stratified test set. Threshold-independent metrics (ROC-AUC,
PR-AUC, Brier) are the headline numbers; operating-point metrics depend on the
chosen threshold.

### `delay`

Test set: 19,294 orders · positive base rate 8.1%.

| Metric | Model | Baseline |
|---|---|---|
| ROC-AUC | **0.7845** | 0.5 |
| PR-AUC | **0.2946** | 0.081 (prevalence) |
| Brier (calibrated) | 0.0655 | — |
| Brier (uncalibrated) | 0.156 | — |

| Operating point | Precision | Recall | F1 | Accuracy |
|---|---|---|---|---|
| Threshold 0.50 | 0.600 | 0.050 | 0.092 | 0.920 |
| Tuned threshold 0.1529 | 0.276 | 0.475 | 0.349 | 0.856 |

Baseline (predict majority class): accuracy 0.919, recall on positives 0.0.

**Top SHAP drivers:** `estimated_delivery_days`, `purchase_month` (seasonality),
`customer_seller_distance_km`, São Paulo customer origin, `approval_delay_hours`,
`cross_state_shipment`, and `total_freight` — all operationally intuitive. See
`reports/shap_delay.png`.

## Interpretation and limitations

- Both models rank substantially better than chance (ROC-AUC ≈ 0.78 and 0.76 vs.
  0.50) and well above the no-skill PR line (PR-AUC ≈ 3.6× and 3.8× prevalence).
- The classes are imbalanced, so the default 0.50 threshold yields low recall on
  the positive class. The API therefore uses the validation-tuned threshold,
  which trades precision for recall to surface more at-risk orders.
- Probabilities are calibrated, so a predicted 0.30 corresponds to roughly a 30%
  empirical rate. This is what makes the scores usable for ranking and triage.
- The dataset is historical Olist data (2016–2018) from the Brazilian market;
  metrics describe that distribution and would need revalidation before use on a
  different retailer or time period.
- The `is_complaint` proxy label defined in the pipeline is a documented
  approximation (the dataset has no returns table) and is not currently served as
  a model.
